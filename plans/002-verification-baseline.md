## Steps

### Step 1: Add Vitest + Testing Library

Run: `bun add -d vitest @vitest/coverage-v8 jsdom @testing-library/react @testing-library/jest-dom @testing-library/user-event`

Add to `package.json` scripts: `"test": "vitest run"` and `"test:watch": "vitest"`.

Create `vitest.config.ts` at repo root:

```ts
import path from "node:path";
import { defineConfig } from "vitest/config";

export default defineConfig({
  resolve: {
    alias: {
      "@": path.resolve(import.meta.dirname, "src"),
    },
  },
  test: {
    environment: "jsdom",
    include: ["src/**/*.test.{ts,tsx}"],
  },
});
```

(The `@/*` alias mirrors `tsconfig.json`. If `tsconfig.json` also maps `collections/*`, do NOT add that alias — tests must not import generated collections; see Step 3.)

**Verify**: `bun run test` → Vitest runs, reports "no test files found" (exit code may be 1 — acceptable until Step 3) or passes trivially.

### Step 2: Add React JSX support for component tests

Vitest needs React JSX transform for `.tsx` tests. Add `@vitejs/plugin-react`: `bun add -d @vitejs/plugin-react` and in `vitest.config.ts` add `plugins: [react()]` with `import react from "@vitejs/plugin-react";`.

**Verify**: `bun run typecheck` → exit 0 (the new config typechecks).

### Step 3: Characterization tests for blog utilities (`src/lib/blog.test.ts`)

`src/lib/blog.ts` imports the generated `collections/server` module at top level, which will fail to resolve under Vitest. Handle it this way, in order of preference:

1. Mock it: `vi.mock("collections/server", () => ({ blogPosts: [] }))` at the top of the test file, then import the functions under test. If the alias cannot resolve at all (Vitest may error before the mock applies), add a `resolve.alias` entry pointing `collections/server` to a small stub file `src/lib/__test-stubs__/collections-server.ts` exporting `blogPosts = []`.
2. **Fallback (only if 1 fails)**: split the pure functions — move `formatBlogDate` and the reading-time word-count math into pure exports in `src/lib/blog.ts` that take plain values (no loader types), keeping existing exports' behavior identical. Keep the diff minimal.

Tests to write (characterize current behavior — do not "fix" anything):

- `formatBlogDate("2026-07-10")` → `"July 10, 2026"` (UTC — also assert a date that would shift across timezones, e.g. `"2026-01-01"` → `"January 1, 2026"`).
- Reading time: construct a post-like object pointing at a real file in `content/blog/` (e.g. `outing-inquiries.md` with `post.path` set to its basename) and assert the result matches `/^\d+ min read$/`; additionally assert the 220-wpm rounding rule using a tiny fixture written to a temp dir is NOT needed — testing against one real content file is sufficient and avoids fixtures.
- Sorting: with `getBlogPosts` this requires the loader — skip `getBlogPosts` unless approach 2 was taken; instead assert the comparator behavior indirectly only if the pure split (fallback) happened. Otherwise omit — do not force it.

**Verify**: `bun run test` → all tests pass.

### Step 4: Smoke test for the audio-player provider (`src/components/ui/audio-player.test.tsx`)

Before writing, read `src/components/ui/audio-player.tsx` fully. Stub media methods at the top of the file:

```ts
window.HTMLMediaElement.prototype.play = vi.fn().mockResolvedValue(undefined);
window.HTMLMediaElement.prototype.pause = vi.fn();
window.HTMLMediaElement.prototype.load = vi.fn();
```

Write render-level tests (model on standard Testing Library patterns; there is no existing test to copy):

- `AudioPlayerProvider` renders children and an `<audio>` element.
- A component calling `useAudioPlayer()` inside the provider can call `play({ id: "x", src: "/audio/sample-call-booking.m4a" })` without throwing, and `pause()` afterwards without unhandled rejection (wrap in `act`).
- `useAudioPlayer()` outside a provider throws (if the hook guards this) — read the source first; only assert what the code actually does.

**Verify**: `bun run test` → all pass, no unhandled-rejection warnings in output.
